4. Isolating the Bottleneck: KV Cache Quantization

A common assumption in the quantization community is that weight quantization is the primary source of quality loss, while KV cache quantization is relatively harmless. We initially shared this assumption. Our Q4_0 and q8_0 runs scored identically (5/10), which seemed to confirm that KV cache quantization was "lossless." However, a deeper review of the literature reveals a more complex picture, and one that matters considerably for production deployments.

What the Research Actually Shows

The impact of KV cache quantization depends heavily on three factors: bit-width, context length, and model architecture. At 8-bit precision, the evidence overwhelmingly shows that KV cache quantization is nearly free:

Q8 KV cache costs +0.05% perplexity vs FP16, effectively free quality. HumanEval degradation: 0 to -0.2 points. -- TechPlained: KV Cache Quantization: Q8 vs FP16 (and Q4 Pitfalls), benchmarking Qwen 3.5 9B at 32K context
FP8 KV cache introduces at most 1-2 points of accuracy degradation across reasoning benchmarks, with lowest recovery at 97% (GPQA:Diamond). For Qwen3.5-27B, accuracy impact was at most 0.7 points. -- vLLM Blog: The State of FP8 KV-Cache and Attention Quantization in vLLM, April 2026

But at 4-bit, the picture changes. Quality loss becomes measurable, and two patterns emerge that are easy to miss in short-context benchmarks:

Pattern 1: Value cache is more sensitive than key cache. Q4 V-cache costs approximately 1.4% perplexity and 1.5+ points on reasoning benchmarks, while Q4 K-cache costs only about 0.4%. Our setup quantizes both to Q4_0, meaning we absorb the full impact of the more sensitive V-cache quantization.
Q4 K+V cache on Qwen 3.5 9B: +2.1% perplexity, -2.8 HumanEval points. The optimal strategy for tight VRAM: Q4 K-cache + Q8 V-cache is the magic combo. -- TechPlained: KV Cache Quantization
Key cache is more sensitive to quantization than value cache. Quantization error in keys has multiplicative dependencies on query vectors, attention scores, AND value differences, while value cache error only depends on the attention score. -- QAQ: Quality Adaptive Quantization for LLM KV Cache (ICCV 2025 Workshop)
Pattern 2: Error accumulates with context length. Unlike weight quantization, where quality loss is fixed regardless of input length, KV cache quantization errors compound as more tokens are processed. This makes short-context benchmarks (like our 10 SWE challenges) a poor proxy for long-context performance.
During each decoding step, the generated token needs to access the quantized low-precision KV Cache, leading to significantly large cumulative quantization errors. Directly applying existing KV cache quantization to long-CoT LLMs causes significant performance degradation. -- PM-KVQ: Progressive Mixed-precision KV Cache Quantization for Long-CoT LLMs (OpenReview)
4-bit KV cache showed approximately 8% accuracy loss on 20-iteration agentic reasoning tasks (89.1% to 81.2%), and 2-bit showed approximately 16.5% loss. -- DASRoot: Is KV Cache Quantization Sabotaging Your Context?

Perhaps the most striking evidence comes from a direct llama.cpp bug report:

Llama-3.3-70B with Q4_K_M weights and q8_0 KV cache produced systematically wrong date calculations. All 3 runs with q8_0 KV were wrong; all 3 runs with f16 KV were correct. -- llama.cpp Issue #10697: compression of q8/q4 cache-type-k cache-type-v badly degrading quality

Why Our Benchmark Did Not Catch This

Our SWE challenges are single-turn tasks with moderate context requirements (2,000 to 16,000 tokens of output). At these lengths, the cumulative error from Q4_0 KV cache remains within noise. The Q4_0 and q8_0 runs scored 5/10 each, with variance on different challenges (C03 vs C08), which we correctly attributed to sampling randomness at temp=0.6.

However, this does not mean KV cache quantization is lossless. It means our benchmark was not designed to detect it. The localbench.substack.com KL divergence study offers a clearer picture:

Qwen 3.6 27B at q4_0 showed KL divergence of 0.581 on long documents specifically, vs 0.087-0.117 overall. Gemma 4 26B-A4B jumped to KL divergence of 1.088 at q4_0. -- localbench: Gemma 4 and Qwen 3.6 with q8_0 and q4_0 KV cache: KL divergence results

The Hybrid Architecture Advantage (and Its Limits)

Qwen3.6 does benefit substantially from its hybrid DeltaNet architecture. Only 10 of 40 layers use standard multi-head attention and maintain a KV cache. The remaining 30 layers use Gated DeltaNet (linear attention with fixed-size recurrent state), which is completely unaffected by KV quantization. This structural property provides a genuine buffer:

q4_0 KV cache quantization achieves perfect lossless performance on hybrid models like Qwen3.5-9B, with BLEU 1.000 across 10 test configurations at 4x compression. Token-level identical output between f16 and q4_0 KV cache. -- llama.cpp Issue #21385: Per-head adaptive KV cache quantization
The linear/sliding attention layers act as error correction. Quantization noise in the few attention layers is absorbed by the surrounding layers. -- llama.cpp Issue #21385

This is a real advantage. On standard transformer architectures (where all layers use full attention), Q4_0 KV cache causes measurable degradation. On Qwen3.6's hybrid architecture, the impact is substantially reduced. But "substantially reduced" is not "zero," particularly at long contexts where the cumulative error in those 10 attention layers still accumulates.

Revised assessment: Q4_0 KV cache on Qwen3.6 is near-lossless at short to moderate contexts (under 32K tokens) thanks to the hybrid architecture. At longer contexts (64K+), the error accumulation in the 10 attention layers becomes measurable, though still significantly smaller than on standard transformers. For maximum quality at long contexts, --cache-type-k q4_0 --cache-type-v q8_0 is the recommended compromise, preserving most of the VRAM savings while protecting the more sensitive value cache.

5. Weight Quantization: The Actual Culprit

While KV cache quantization plays a role that we initially underestimated, the dominant source of our quality gap remains weight quantization. At comparable bit-widths, weight quantization consistently causes larger accuracy drops than KV cache quantization:

LLaMA-2-13B with W4 (weight-only 4-bit) had perplexity 4.99 vs 4.88 FP16 baseline. Adding KV4 on top (W4KV4) only moved it to 5.00. The KV cache quantization added only 0.01 perplexity on top of weight quantization's 0.11. -- WKVQuant: Quantizing Weight and Key/Value Cache for Large Language Models Gains More
On Llama-3.1-8B-Instruct, Q3_K_S weight quantization dropped MMLU from 63.50% to 59.31% (-4.19 points) and GSM8K from 77.63% to 68.31% (-9.32 points). These are far larger drops than any KV cache quantization result at similar compression ratios. -- Which Quantization Should I Use? A Unified Evaluation of llama.cpp Quantization

Our original setup used IQ3_XXS (approximately 3-bit), which sits below the 4-bit threshold. Upgrading to IQ4_XS improved SWE challenge scores from 5/10 to 9/10 (with /no_think mode), directly confirming that weight quantization was the primary bottleneck:

The Compounding Effect

What makes our configuration particularly aggressive is that we stack two forms of quantization simultaneously: IQ3_XXS weights and Q4_0 KV cache. The research from WKVQuant cited above suggests these losses are roughly additive. On a standard transformer, that would mean absorbing both the weight quantization penalty (the larger factor) and the KV cache quantization penalty (the smaller but non-trivial factor). The hybrid architecture provides a partial shield against the KV component, but it does not protect against weight quantization at all.

The practical improvement confirms this analysis: upgrading weights from IQ3_XXS to IQ4_XS (crossing the 4-bit threshold) improved scores from 5/10 to 7/10. Adding asymmetric KV cache (Q4 keys + Q8 values), proper inference settings (temp=0.6, retries), and /no_think mode for code generation delivered the final configuration, closing the gap with cloud-served Gemma 31b to just 1 point (9/10 vs 10/10).

6. Three Configuration Pitfalls and Their Fixes

Our initial tests scored 3/10, with 5 challenges producing no visible answer. After identifying three configuration errors, we improved to 5/10 with all 10 challenges answered. Each issue is independently documented by the broader community.

Pitfall 1: Thinking Token Overflow

Symptom: Model spends all max_tokens in <think> blocks. API returns "content": "" with all tokens consumed by reasoning_content.
Impact: 5 of 10 challenges returned NO ANSWER.
Fix: --reasoning-budget 4096 on the server, or per-request "thinking_budget_tokens": 4096.

This is the most widely reported Qwen3 operational issue, with 8+ independent bug reports:

I can't be the only one can I? -- HuggingFace Discussion #19, user phoebdroid, on endless reasoning loops
When posing straightforward reasoning questions to Qwen3-Thinking models, the system enters an endless cycle where the correct answer repeats continuously without terminating inference. -- LM Studio Bug #1018, user godmonero
Output from openai client only contains generated text in the reasoning field while the content field, which should contain the model answer, is always None. -- vLLM Issue #38894

The --reasoning-budget fix has moved from an obscure workaround to a standard recommendation. A community benchmark on Qwen3 9B HumanEval showed: full reasoning = 94%, hard budget cutoff = 78%, budget with transition message = 89%.

Pitfall 2: Temperature 0.0 Degradation

Symptom: Repetition loops, performance degradation, stuck generation.
Fix: Use temperature=0.6, top_p=0.95, top_k=20 (official recommendation).
DO NOT use greedy decoding, as it can lead to performance degradation and endless repetitions. -- Official Qwen3 Model Cards (consistent across all variants)
Greedy decoding kills its output ... stick to sampling. -- Jan.ai: Best Settings to Run Qwen3

Pitfall 3: Tool Calling Instability

Symptom: Function calling fails with wrong arguments, infinite loops, malformed JSON.
Partial Fix: Enable preserve_thinking, raise output token limits, use --swa-full flag.
Status: Partially unresolved. Affects even BF16 precision.
The tool use has been absolutely impossible to use in an agentic mode with LM Studio and OpenCode. It loops non-stop sending the wrong arguments for file writes. -- HuggingFace Discussion #51
Tool call parsing fails intermittently across all quantization levels including BF16. Appears architectural or deeply rooted in model/inference engine interaction. -- Unsloth GGUF Discussion #5

8. Failure Analysis: Where IQ4_XS Falls Short

Of the 10 SWE challenges, IQ4_XS failed three that Gemma 31B passed (C07, C09) plus one universal failure (C06). Examining the actual outputs reveals distinct failure modes with actionable fixes.

Token Economy: A Structural Difference

IQ4_XS uses 4 to 14 times more tokens than Gemma across all challenges. This is driven by the extended thinking architecture. On challenges where both pass, this is merely a cost difference. On failures, the excess tokens signal that the model is struggling:

Token RangeIQ4_XS Outcome
Under 5,000 tokens5 PASS, 0 FAIL
5,000 to 6,000 tokens2 PASS, 2 FAIL
Over 6,000 tokens0 PASS, 1 FAIL

Every IQ4_XS challenge under 5,000 tokens passes. Every one over 6,000 fails. Gemma averages 650 tokens per challenge vs IQ4_XS's 5,600.

C09: Async Queue Bugs (IQ4_XS FAIL, Gemma PASS)

IQ4_XS consumed all 16,384 tokens on both attempts. The output is not code but a verbose markdown analysis. The model correctly identifies the fix (replace asyncio.Event with asyncio.Condition) but cannot stop explaining and start coding:

*   **Explanation:** The method calls `self.not_empty.set()`. While `Event.set()`
    is synchronous... Wait, checking the documentation... `Event.set()` is synchronous.
    So this isn't a syntax error.
*   **Re-evaluating "At least 4 bugs"**:
    Let's look at the logic flow again.

Gemma produced 1,193 tokens of clean, working code on its first attempt. No explanation, just the corrected implementation. The --reasoning-budget flag caps the <think> block, but it does not prevent the model from reasoning in its visible output.

Fix applied: /no_think mode disables the thinking phase entirely, combined with a system prompt that constrains output to a single code block. The --reasoning-budget-message flag provides a graceful transition when the budget is hit. The preserve_thinking template option prevents re-reasoning loops in multi-turn contexts.

C07: Mini Regex Engine (IQ4_XS FAIL, Gemma PASS)

IQ4_XS builds a recursive descent parser that converts regex patterns into an AST stored as tuples: (type, args). The parser is well-structured. The bug is at the parser-matcher boundary: concat nodes store child indices as a nested tuple, but the matcher tries to use that tuple as a list index:

TypeError: list indices must be integers or slices, not tuple

Gemma avoids this entirely with an object-oriented design where each node type is a separate class with its own match() method. No index arithmetic, no type confusion possible. Both models needed 2 attempts, but Gemma's second attempt used a fundamentally cleaner architecture.

This pattern is characteristic of quantized models: the high-level algorithm is correct, but fine-grained data structure consistency degrades. The model "knows" both the parser and the matcher, but the seam between them is where quantization-induced imprecision manifests.

C06: Calendar Interval Merging (Test Bug Found and Fixed)

The original C06 test contained a bug: it asserted that free_slots(0, 500, min_duration=150) should return [(200, 300)], but that slot is only 100 units long, which is less than the min_duration=150 filter. Every model correctly implemented the filter and returned an empty list, but the test expected the wrong answer.

After fixing the test to use asymmetric free slots (50, 200, and 100 units), the results diverge by mode: IQ4_XS /no_think and Gemma both pass on the first attempt, while IQ4_XS with thinking and IQ3_XXS both fail on the priority resolution logic. The thinking mode appears to overcomplicate the priority trimming, producing bugs in edge cases around higher-priority event insertion. Direct code generation handles this cleaner.

Cross-Cutting: Architectural Choices Under Quantization

Gemma consistently uses simpler, more modular architectures (OOP classes, standard library primitives). IQ4_XS tends toward clever but fragile designs (tuple-packed ASTs, manual state management). Under quantization pressure, the simpler architecture is more robust because there are fewer opportunities for subtle inconsistencies. This is not unique to quantization, but quantization amplifies the effect.

9. Fixing the Thinking Overflow

Qwen3.6's extended thinking architecture is its greatest strength and its most dangerous failure mode. The model can reason deeply about complex problems, but it can also spiral into endless deliberation. Four server-side and prompt-side fixes address this:

Fix 1: /no_think Mode

Append /no_think to the user message to disable the thinking phase entirely. For code generation tasks with clear specifications, direct code output is more reliable than reasoning-then-coding:

{"role": "user", "content": "Implement quicksort in Python /no_think"}

Combine with a system prompt that constrains output format:

{"role": "system", "content": "You are a code generator. Respond with only the requested code in a single Python code block. No explanations."}
Non-thinking mode scored 88% on HumanEval vs 94% with full thinking and 78% with hard budget cutoff. A small quality tradeoff for dramatically improved output discipline. -- r/LocalLLaMA: Reasoning Budget Controls Analysis

Fix 2: --reasoning-budget-message

Instead of hard-cutting when the thinking budget is exhausted, inject a transition message that cues the model to produce its answer:

--reasoning-budget 4096 \
--reasoning-budget-message "I need to provide my answer now."

This recovers most of the quality lost from hard cutoff (89% vs 78% on HumanEval) while preventing infinite thinking loops.

Fix 3: preserve_thinking

Enable via --chat-template-kwargs '{"preserve_thinking":true}'. This retains the model's reasoning in conversation history so subsequent turns can reference prior thinking. Without it, the model re-derives everything from scratch, increasing total token output. Multiple community reports identify this as the single most impactful fix for Qwen3.6 reasoning loops.

Enabling preserve_thinking fixed tool use failures and endless reasoning loops with Qwen3.6-35B-A3B. -- HuggingFace Discussion #51

Fix 4: Jinja Templating + DeepSeek Reasoning Format

Use --jinja to enable the Jinja2 chat template embedded in the GGUF, and --reasoning-format deepseek to separate thinking content from visible output. This ensures thinking tokens appear in reasoning_content rather than polluting the response:

--jinja \
--reasoning-format deepseek \
--chat-template-kwargs '{"preserve_thinking":true}'

Recommended Server Configuration

llama-server \
    -m models/Qwen3.6-35B-A3B-UD-IQ4_XS.gguf \
    -ngl 99 --n-cpu-moe 30 \
    --flash-attn on \

The Constraint: Quantization Degrades Agent Reliability

8GB models are excellent for generating code snippets, short utility functions, and boilerplate scaffolding. But if your goal is true agentic coding, multi-file edits, autonomous debugging loops, architectural refactoring, this weight class will hit its intelligence ceiling. -- LocalLLM.in: Best Local LLMs for 8GB VRAM
A reliability peak occurs for 4-bit quantized models, indicating that quantizing moderately sized base models offers the best reliability-efficiency trade-off, though performance below 4 bits becomes increasingly problematic. -- OpenReview: Reliability Scaling Laws for Quantized LLMs

At IQ3_XXS, these concerns are well-founded. However, upgrading to IQ4_XS crosses the 4-bit threshold: SWE challenge scores improved from 5/10 to 9/10 (with /no_think mode), and the model now handles challenges (C06 Calendar, C07 Regex, C08 Hash Ring, C09 Async) that IQ3_XXS consistently failed. For single-turn and short-context agent tasks, IQ4_XS is a credible option. For extended multi-turn agent loops, the remaining quantization impact and KV cache error accumulation may still be limiting factors.

12. Final Verdict

9/10: One point from cloud Gemma, running on a laptop GPU.

Qwen3.6-35B-A3B at IQ4_XS with /no_think mode scores 9/10 on hard SWE challenges, one point behind cloud-served Gemma 4 31B (10/10). The only remaining failure (C04: Tree Serialization) requires extended reasoning that /no_think disables. This is a 35B-parameter model running on a laptop GPU with 128K context, delivering quality that was previously only available through cloud APIs.

The journey from 3/10 to 9/10 demonstrates that configuration matters as much as raw model capability. The six levers were: fixing inference settings (temp=0.6, reasoning budget), crossing the 4-bit weight quantization threshold (IQ3_XXS to IQ4_XS), using asymmetric KV cache (Q4 keys + Q8 values), disabling thinking for code generation tasks (/no_think), fixing a test bug in C06, and adding retries for sampling variance.

One tradeoff emerged: C04 (Tree Serialization) passes with thinking enabled but fails with /no_think. The optimal strategy is task-dependent: use /no_think for straightforward code generation, enable thinking for complex reasoning tasks. A hybrid approach that selects the mode based on task type would achieve 10/10, matching cloud Gemma entirely.

What We Established

  • IQ4_XS + /no_think scores 9/10, up from 3/10 with the original misconfigured setup, a 200% improvement through configuration alone
  • The gap to cloud Gemma is just 1 point (9/10 vs 10/10), with the only failure (C04) being a task that genuinely requires extended reasoning
  • Disabling thinking for code tasks is a net positive: +1 point (C07, C09 recovered), -1 point (C04 regressed), but the recovered challenges were previously hard failures while C04 is sampling-sensitive
  • Weight quantization is the dominant quality factor, but KV cache quantization contributes cumulative errors at longer contexts, particularly in the value cache
  • Asymmetric KV cache (Q4 keys + Q8 values) is the optimal compromise, protecting the 3.5x more sensitive value cache while preserving VRAM savings
  • A hybrid thinking strategy would achieve 10/10: use /no_think for code generation, enable thinking for complex reasoning (C04), matching cloud Gemma entirely
  • Token efficiency matters: /no_think uses 735-5,336 tokens per challenge vs 3,336-16,384 with thinking. Every challenge under 5,000 tokens passes
  • The full-precision model is exceptional: SWE-bench 73.4%, double Gemma 4 on tool use
