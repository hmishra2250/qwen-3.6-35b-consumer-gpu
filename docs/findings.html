4. Full Model vs Quantized: What We Lost

The full Qwen3.6 dominates Gemma 4 on coding. Our quantized version losing to Gemma is entirely a quantization story, not a model quality story.

IQ3_XXS is approximately 3-bit weight quantization. Research shows this causes significant quality degradation:

Most of the original model’s advantages are lost at 3-bit precision. Qwen3 shows more pronounced performance degradation under low-bit quantization than LLaMA3. — arXiv:2505.02214, Qwen3 quantization study, May 2025
On agentic coding loops, the delta between Q4 and IQ3 is most visible as occasional bracket mismatches and weaker tool-call formatting. — Compute Market: Qwen3.6 Local Hardware Guide, April 2026

A specific Qwen3.5-122B study found even more severe degradation:

The Q3-level quantization exhibited unexpectedly low quality, with a high probability of partial random garbled output and infinite repetition when encountering tasks like tool calls and Python code generation. — HuggingFace Discussion #7, user lingyezhixing

5. Three Critical Pitfalls (and Fixes)

Our initial tests scored 3/10 with 5 challenges producing no visible answer. After discovering three configuration errors, we improved to 5/10 with all 10 challenges answered. Every issue we found is independently documented by the community.

Pitfall 1: Thinking Token Overflow

Symptom: Model spends all max_tokens in <think> blocks. API returns "content": "" with all tokens in reasoning_content.
Impact: 5/10 challenges returned NO ANSWER.
Fix: --reasoning-budget 4096 on the server, or per-request "thinking_budget_tokens": 4096.

This is the most widely reported Qwen3 issue, with 8+ independent bug reports:

I can’t be the only one can I? — HuggingFace Discussion #19, user phoebdroid, on endless reasoning loops
When posing straightforward reasoning questions to Qwen3-Thinking models, the system enters an endless cycle where the correct answer repeats continuously without terminating inference. — LM Studio Bug #1018, user godmonero
Output from openai client only contains generated text in the reasoning field while the content field, which should contain the model answer, is always None. — vLLM Issue #38894
OpenClaw repeats the same sentences and fails until context runs out. — HuggingFace Discussion #20, user zeerd

The --reasoning-budget fix has grown from an obscure workaround to a standard recommendation. A community benchmark on Qwen3 9B HumanEval showed: full reasoning = 94%, hard budget cutoff = 78%, budget with transition message = 89%.

Pitfall 2: Temperature 0.0 Degradation

Symptom: Repetition loops, performance degradation, stuck generation.
Fix: Use temperature=0.6, top_p=0.95, top_k=20 (official recommendation).
DO NOT use greedy decoding, as it can lead to performance degradation and endless repetitions. — Official Qwen3 Model Cards (consistent across all variants)
Greedy decoding kills its output … stick to sampling. — Jan.ai: Best Settings to Run Qwen3

Pitfall 3: Tool Calling Instability

Symptom: Function calling fails with wrong arguments, infinite loops, malformed JSON.
Partial Fix: Enable preserve_thinking, raise output token limits, use --swa-full flag.
Status: Partially unresolved — affects even BF16 precision.
The tool use has been absolutely impossible to use in an agentic mode with LM Studio and OpenCode. It loops non-stop sending the wrong arguments for file writes. — HuggingFace Discussion #51
Tool call parsing fails intermittently across all quantization levels including BF16. Appears architectural or deeply rooted in model/inference engine interaction. — Unsloth GGUF Discussion #5

7. Q4_0 KV Cache: Confirmed Lossless

Our finding confirmed: Q4_0 and q8_0 KV cache scored identically (5/10) with the same error patterns. The hybrid DeltaNet architecture makes KV quantization essentially free.
q4_0 KV cache quantization achieves perfect lossless performance on hybrid models like Qwen3.5-9B, with BLEU 1.000 across 10 test configurations at 4x compression. Token-level identical output between f16 and q4_0 KV cache. — llama.cpp Issue #21385
The linear/sliding attention layers act as error correction — quantization noise in the few attention layers is absorbed by the surrounding layers. — llama.cpp Issue #21385

Why this works: Qwen3.6-35B-A3B uses Gated DeltaNet (linear attention, no KV cache) for 30/40 layers. Only 10 layers use standard multi-head attention with KV cache. Q4_0 quantization affects just those 10 layers — a 75% reduction in impact compared to a standard transformer. This enables 128K context in 8 GB VRAM with zero quality loss.

8. Viability for Local Agent Orchestration

The Good News

The full-precision Qwen3.6-35B-A3B is arguably the best open model for agent tasks:

Qwen3.6 wins — and sometimes it isn’t close on coding metrics, with MCPMark showing Qwen more than twice as capable at integrating with MCP tools. — BuildFastWithAI: Qwen3.6 Review
Qwen 3.6 demonstrates superior reliability for complex task execution, while Gemma 4 often struggled to trigger the correct tools. — Gemma 4 Wiki: Qwen 3.6 vs Gemma 4

On the Aider Polyglot benchmark, Qwen3.6-35B-A3B at Q4_K_M scored 78.67% (177/225) on an RTX 5070 8 GB — a +33 point improvement over Qwen3.5-9B.

The Bad News: Quantization Degrades Agent Reliability

8GB models are excellent for generating code snippets, short utility functions, and boilerplate scaffolding. But if your goal is true agentic coding — multi-file edits, autonomous debugging loops, architectural refactoring — this weight class will hit its intelligence ceiling. — LocalLLM.in: Best Local LLMs for 8GB VRAM
A reliability peak occurs for 4-bit quantized models, indicating that quantizing moderately sized base models offers the best reliability-efficiency trade-off, though performance below 4 bits becomes increasingly problematic. — OpenReview: Reliability Scaling Laws for Quantized LLMs

At IQ3_XXS, our model sits below the 4-bit reliability threshold. Tool calling, structured output, and multi-step reasoning all degrade. The model is usable for single-turn coding tasks but unreliable for autonomous agent loops.

10. Final Verdict

Usable for coding assistance. Not reliable enough for autonomous agents.

Qwen3.6-35B-A3B at IQ3_XXS on 8 GB VRAM is a remarkable technical achievement — a 35B-parameter model running on a laptop GPU with 128K context. It handles single-turn coding tasks respectably (5/10 on hard challenges) and the Q4_0 KV cache is genuinely lossless.

But for agent orchestration (OpenClaw, Hermes, Smolagents), the 3-bit weight quantization drops quality below the reliability threshold. Tool calling is unstable even at full precision, and it gets worse at IQ3_XXS. The thinking overflow issue — now mitigated with --reasoning-budget — was a showstopper before the fix was discovered.

For serious agentic use on consumer hardware: either upgrade to Q4_K_M quantization with CPU offload (same model, better quality, same speed), or use Qwen3.5-9B at Q4_K_M for fully on-GPU inference at 55 t/s with near-perfect quality retention.

What We Proved

  • Q4_0 KV cache is free on hybrid architectures — 128K context with zero quality loss
  • Reasoning budget is mandatory — without it, 50% of coding tasks produce no answer
  • Temperature=0 is actively harmful for Qwen3 — causes repetition loops
  • IQ3_XXS is below the 4-bit reliability threshold for agent tasks
  • The full model is exceptional — SWE-bench 73.4%, double Gemma 4 on tool use
